# Kiosk Admin — Constitution

Project-wide rules that apply to every spec, plan, task, and code change.
These are immutable unless explicitly revised here first.

---

## Agent Workflow

### Orientation (start every session here)
1. Read `steering/product.md`, `steering/tech.md`, `steering/structure.md` — always-relevant context.
2. Check the **Implementation status** section in `steering/structure.md` to see what is done and what is next.
3. Read the relevant spec folder before touching any code: `specs/00N-feature-name/requirements.md`, then `design.md`, then `tasks.md`.

### Spec-Driven Development (SDD)
This project follows SDD — specs are written first and code implements them, not the reverse.
- **Never create a feature without a spec.** If a spec is missing, write it before writing code.
- The spec is the source of truth. If code and spec disagree, fix the code.
- Each spec folder has: `requirements.md` (EARS-style requirements), `design.md` (interfaces and data flow), `tasks.md` (concrete implementation checklist).
- Mark tasks complete in `tasks.md` as you finish them.

### Common commands (run from `kiosk-admin/`)
```bash
npm run dev          # start dev server on :3000 (Turbopack)
npm run build        # production build — must pass before any PR
npm run lint         # ESLint check
npx tsc --noEmit    # type-check without building

# Prisma
npx prisma migrate dev --name <name>   # create + apply a migration (SQLite/PG)
npx prisma db push                      # sync schema without migration (MongoDB)
npx prisma generate                     # regenerate client after schema changes
npx prisma studio                       # browse the database
```

### After making changes
- Run `npx tsc --noEmit` from `kiosk-admin/` — fix all type errors before moving on.
- Run `npm run build` to confirm no build-time errors.
- If you add a new Shadcn component: `npx shadcn@latest add <name>` (from `kiosk-admin/`).
- If you change `prisma/schema.prisma`: run `prisma migrate dev` then `prisma generate`.

### Commit message format
```
feat(scope): short description, refs specs/00N-feature-name
fix(scope): short description
```
Example: `feat(devices): add tag filtering, refs specs/002-device-management`

### What NOT to do
- Do not add `any` — use `unknown` + narrowing or proper types.
- Do not call device REST APIs directly from components or pages — always go through `lib/providers/`.
- Do not put secrets or API keys in client-side code.
- Do not edit `lib/generated/` — it is auto-generated by Prisma.
- Do not create `middleware.ts` — this project uses `proxy.ts` (Next.js 16 convention).
- Do not add a new npm dependency without a comment in code explaining why existing deps can't meet the need.

## Language & Runtime
- The system shall use TypeScript with `strict: true` everywhere — no `any`, no exceptions.
- The system shall target Node.js 20 LTS.
- The system shall use ES modules (`"type": "module"` in package.json).

## Framework
- The system shall use Next.js App Router exclusively — no Pages Router.
- The system shall keep all server-side logic in Server Components, Server Actions, or API Route Handlers. No client-side secrets.
- The system shall use Shadcn/ui components as the baseline UI library. Custom components extend Shadcn; they do not replace it.

## Database
- The system shall default to SQLite via Prisma when `DATABASE_URL` is not set.
- The system shall support PostgreSQL and MongoDB as opt-in alternatives via `DATABASE_PROVIDER` env var.
- The system shall never perform raw SQL or direct DB calls outside of `lib/db.ts` and Prisma models.
- The system shall run `prisma migrate dev` (relational) or `prisma db push` (MongoDB) — never edit the DB schema by hand.

## Security
- The system shall encrypt all device passwords at rest using AES-256-GCM before writing to the database.
- The system shall never expose device IP addresses, ports, or passwords in any API response sent to the browser.
- The system shall validate every authenticated API route — unauthenticated requests receive `401`.
- The system shall validate every admin-only API route for `ADMIN` role — `VIEWER` requests receive `403`.
- The system shall sanitise and validate all user-supplied input at API route boundaries using Zod.

## Provider Abstraction
- The system shall route all device communication through a `KioskProvider` implementation. No component or route shall call a device REST endpoint directly.
- The system shall gate UI controls by the provider's static `capabilities` flags. Controls for unsupported capabilities shall not render.

## Real-time
- WHERE MQTT is configured, the system shall use MQTT + SSE for live device updates.
- WHERE MQTT is not configured, the system shall fall back to client-side HTTP polling on a 30-second interval.

## Testing
- The system shall generate tests from specs, not the other way around.
- The system shall maintain ≥ 80% coverage on `lib/` modules.
- The system shall use Vitest for unit and integration tests, and Playwright for e2e tests.
- The system shall mock all outbound HTTP calls to devices using msw in tests.

## Code Style
- The system shall use named exports — no default exports except Next.js page/layout files.
- The system shall keep files under 300 lines. Files exceeding this shall be split.
- The system shall co-locate types with the module that owns them; shared types live in `lib/types.ts`.
- The system shall not introduce a new `npm` dependency without a comment explaining why an existing dependency cannot meet the need.

## Commits & PRs
- The system shall reference the spec in every commit: `feat(devices): add tag filtering, refs specs/002-device-management`.
- The system shall not merge a PR whose tests are failing or whose coverage regresses.

## Out of Scope (this project will NOT)
- Provide a native mobile app.
- Manage devices via Fully Cloud API (only direct REST to device).
- Support multi-tenant / multi-organisation setups.
- Expose a public API for third-party integrations.
